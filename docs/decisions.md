# Decision Register

**This file is the single source of truth for D-numbers.** Every other document references
decisions *by number* rather than restating them. If a decision changes, it changes here first.

Why this exists: D1-D23 were previously scattered across six documents, each paraphrasing them
slightly differently. [`brief.md`](brief.md) drifted out of sync twice as a result — still describing a
Rust agent after D8 had made the project all-Go. Restatement is what rots; references don't.

Rationale lives in the research reports, which are **append-only historical records** and are
never rewritten: [`research-scouting-r1.md`](research-scouting-r1.md) (D1-D9) and
[`research-review-r1.md`](research-review-r1.md) (D10-D23, four adversarial reviewers).

Status: **firm** = settled, changing it is a project pivot · **open** = awaiting evidence or
an owner call · **scoped** = settled but with a named condition that could narrow it.

## Scope and phasing

| # | Decision | Status |
|---|---|---|
| **D1** | **Phase 1 is observe-and-audit only.** Decisions are recorded, not enforced. Enforcement lands in Phase 2 once the classifier's real false-positive rate is known. DLP has no cheap partial-credit MVP: false negatives leak, false positives block legitimate work. | firm |
| **D49** | **Enforcement is POST-DECISION: the engine records a Decision, then dispatches it to a registered enforcer that carries it out, auditing the outcome; observe-only is the default and enforcement CONTAINS, it does not PREVENT (Phase 2 kickoff).** The closed action set (D14) and a proven enforcer contract (T-020) existed but nothing carried out a Decision. Inline fanotify blocking (deny within the permission window) is deferred because the pipeline cannot complete in the 532µs budget (T-002) — classification reads whole files. So the tractable step is post-decision enforcement: `Engine.Enforcers`, invoked AFTER the Decision is recorded (order deliberate — the trail shows what was decided even if enforce fails or the process dies), with the enforcement outcome audited — a failed enforcement is a high-severity entry, never silence (D14). With NO enforcers registered the engine is observe-only (D1); registering one turns enforcement on per action. A real `quarantine` file enforcer moves a flagged file to an owner-only directory. The enforcement TARGET (the file path) is supplied by the engine from the event via a `TargetedEnforcer` interface — the Decision still carries only the verdict (D39's target gap avoided, not widened); the enforcer sees the Decision for the verdict and the target for what to act on, exactly as CrowdSec's decision carries the IP. **Honest bound (D16):** post-decision enforcement contains AFTER detection (the file was already read — that is how it was classified); it does not prevent the access that triggered it, and quarantine is defeatable by root. "Enforcement" is not "prevention". | firm |
| **D2** | **No clipboard events.** Wayland mediates the clipboard through the compositor and hands data only to the focused client — system-wide observation is *architecturally prevented*, not merely hard. Documented as a platform gap. | firm |
| **D7** | **Keep the core small.** Prior OSS DLP died of maintenance economics, not bad detection (MyDLP acquired, OpenDLP abandoned, Apache Metron retired to the Attic). Ambition is the failure mode. | firm |
| **D9** | **Windows/macOS via GitHub Actions runners, not local VMs.** Windows: a VM solves *dev* (test-signing loads unsigned drivers); shipping needs an EV cert (~$300-600/yr) + Microsoft attestation. macOS: Apple's EULA permits virtualisation only on Apple hardware, and Endpoint Security needs an entitlement Apple grants selectively. Not an infrastructure problem. | firm |

## Detection and enforcement

| # | Decision | Status |
|---|---|---|
| **D3** | **fanotify is the primary hook**, with **fail-open engineered from the first commit** (self-PID bypass, response-timeout watchdog, safe teardown). The kernel does not fail open for you — blocked processes park in `TASK_UNINTERRUPTIBLE` and can hang the machine. Blocks open/read/exec; **not** rename or unlink. | firm |
| **D4** | **Decision carries confidence, not certainty.** Classification is noisy by nature (Presidio ~22.7% precision on person names). Policy must consume confidence and thresholds, never a clean boolean. | firm |
| **D5** | **Endpoint classification is patterns and deny-lists only.** No spaCy/NER on the agent — Microsoft deploys Presidio server-side because the NER path is not endpoint-viable. | firm |
| **D14** | **The Enforcer action set is CLOSED and TYPED** (Block / Alert / Quarantine-local / Encrypt-local). An open action framework lets a compromised control plane express "upload file to URL", indistinguishable from normal telemetry. This is what makes "the server coordinates, does not control" architectural rather than aspirational. | firm |
| **D39** | **The USB enforcer acts on the global `authorized_default` posture; per-device enforcement is deferred (T-020).** D1 asked for one real, non-stub enforcer to prove the `Enforcer` contract end to end. `internal/enforcers/usb` implements `core.Enforcer`, advertises ALLOW/BLOCK, and enacts a Decision by writing `/sys/bus/usb/devices/usbN/authorized_default` (the standard Linux mechanism USBGuard builds on) — a real enforcement point. It refuses an action it does not advertise rather than silently no-oping (a no-op is an enforcement that did not happen but looks like it did). The producer (`internal/connectors/usb`) pseudonymises the USB serial at the source with a keyed HMAC — a raw serial is a durable re-identifier and must never enter the event stream (D23). **Honest boundary this surfaced:** the Decision carries `event_id` and the action but NO enforcement TARGET, so this is a GLOBAL posture, not "deauthorise THIS device". Per-device enforcement needs a typed target on the Decision — the small, identifiable core addition D26 predicts a new-shape capability requires — deferred until targeted enforcement is actually built rather than added speculatively. It does NOT test the fail-open/blocking contract (that is the fanotify watchdog's, T-011), and Phase-1 observe-only means nothing in the live pipeline invokes it — the enforcer is real and CAN block; a test drives it. | firm |
| **D34** | **The policy engine runs with a restricted capability set — no network, no clock, no randomness.** OPA/Rego (D6) is instantiated with every nondeterministic builtin and `opa.runtime` removed, filtered by OPA's own `Nondeterministic` flag so an OPA upgrade that adds a new clock or network builtin is excluded automatically. A policy that references `http.send`, `time.now_ns`, `rand.*` fails to LOAD, named, not at eval. This does three things with one mechanism: it makes "the server coordinates, it does not control" a capability boundary rather than a review promise — a distributed policy (Phase 2) cannot reach out regardless of its content; it makes decisions **deterministic and therefore replayable** (D27, `core.Replay`), the precondition for the audit trail being an investigation tool; and it removes `http.send`, an SSRF/exfil primitive, from every endpoint. The Decision's non-deterministic fields (`decision_id`, `decided_at`) are set by the Go layer, outside the policy. The action is mapped through a closed table (D14) — an action the enum does not define is a failed Decision, never a silent ALLOW — and confidence is clamped strictly below 1.0 (D4). OPA lives in `internal/policy`, never core (`check-core-deps.sh`). **Honest limit:** the capability set removes side effects, not CPU; a pathological policy can still be slow, which is a throughput concern because policy runs off the permission-response path in observe-only Phase 1, not a machine-hang. | firm |
| **D17** | **Fail-open is per-policy, and is itself a bypass.** Make classification slow (huge file, zip bomb, ReDoS, load) and every Block becomes an Allow — the starvation trick used against WAFs. Every timeout-allow emits a high-severity audit event; scan budgets capped; timeout *rate* is its own signal. | firm |
| **D43** | **Every terminal pipeline outcome is logged structurally (stdlib `log/slog`) with the event id as correlation id and a stable error CATEGORY (T-028).** Full OTel tracing is cut from Phase 1, but silent-in-the-logs is the milder cousin of silent-in-the-audit and the same rule applies (D17): a failure must be loud and countable. The dispatcher logs decided/failed/timeout/no-decision/not-recorded with `event_id`, `stage`, `kind`, `severity` and `category`, at a slog level matching the severity (a timeout logs at Warn — it turns a Block into an Allow, so a rising rate is greppable). `core.Category(err)` maps errors to stable slugs by IDENTITY (`errors.Is`), not by string, so wrapping does not change a category and a consumer can alert on `category=not_recorded` without parsing prose. **Logs carry ids and categories ONLY, never content** — a log is a wire (D10), asserted by a test that a content marker never appears. The logger is nil-safe (discard default); the wiring point is `Dispatcher.Logger`, set by whatever assembles the pipeline. Correlation id is the event id, not a trace/span id — OTel remains deferred. | firm |
| **D18** | **Phase 1 exercises the fail-open watchdog for real** (injected delay, assert auto-`FAN_ALLOW`), even though verdicts stay always-allow. USB enforcement does *not* test this contract — attach-time allow/deny has no blocked process and no race. **BUILT (T-011):** `internal/agent/watchdog` races evaluation against a hard per-event budget and answers the kernel with `FAN_ALLOW` on timeout, emitting a HIGH-severity audit event; self-PID is allowed by identity before any evaluation (no self-deadlock); the answer precedes the ledger write, and an audit failure surfaces without retracting the allow. The kernel-facing `fanotify_response` write is a thin `Responder` adapter, so the decision logic is fully tested without CAP_SYS_ADMIN; a privileged integration test exercises the real `FAN_OPEN_PERM` path and skips LOUDLY otherwise. Separate mechanism from the dispatcher's `StageDeadline` (which bounds an in-process wait Go cannot preempt) — the watchdog owns the kernel answer itself. | firm |
| **D16** | **Tamper-detection, never tamper-prevention.** Heartbeat / dead-man's-switch / "agent last seen". Anyone with root defeats a host agent; there is no fix that doesn't require distrusting the OS the agent runs on. | firm |
| **D42** | **The dead-man's-switch reports a silent host as OVERDUE, not tampered — a signal for a human, not an accusation (T-018).** This backs D16's honest claim: tamper-DETECTION is only real if the ABSENCE of an agent is detectable. The agent emits a `Heartbeat` (agent_id, sequence, observed_at) on its own subject; the control plane records it so "last seen per host" is a queryable fact that advances whether the agent reported real telemetry or only checked in. `OverdueAgents` is a PURE detector over last-seen times and a threshold — an agent silent past the threshold is overdue — kept pure because the logic that decides "someone should look" must be trivially verifiable. A clean `systemctl stop` is itself recorded (an `ExecStopPost` hook), so the case that STANDS OUT is "silent past threshold with no stop record", not an ordinary operator stop. **Honest limits (D16, D41):** defeatable by root (stop the agent, suppress the heartbeat, mask the unit, block the control plane) and by a compromised control plane (forge a heartbeat); last-seen lives in the non-evidentiary aggregate. Overdue ≠ tampered — legitimate offline periods exist and the offline queue (T-024) heals brief gaps; the value is noticing the COMMON case, not defeating an adversary who owns both ends. The threshold is several heartbeat intervals so normal jitter does not cry wolf. | firm |
| **D33** | **The endpoint detector engine MUST be a linear-time (RE2-class) matcher.** Classification runs on attacker-influenced bytes in the unprivileged worker, so a backtracking regex engine is a denial-of-service primitive — and because slow classification fails open (D17), a pattern driven to catastrophic backtracking turns every Block into an Allow. Go's `regexp` (RE2) cannot be driven to super-linear time, which removes the primitive rather than mitigating it. This is why the classifier (T-007) uses stdlib `regexp` and why introducing a backtracking engine for detection is forbidden, not discouraged. Detectors are format-match plus a validator (CPF/CPF-check-digits, card/Luhn, SSN/structural-only) emitting **type + confidence + count** with confidence never 1.0 (D4) and no content, hash or fingerprint on the wire (D10/D11) — asserted by a wire-byte grep, not prose. | firm |

## Privacy and data handling

| # | Decision | Status |
|---|---|---|
| **D10** | **No exact-match hashing of low-entropy PII.** SSN and Brazilian CPF ~10⁹, cards ~10⁷ after BIN+Luhn, DOB ~36.5k — all brute-forceable, and **salting does not fix it**. Worse, cross-endpoint matching needs a deterministic key, which would have to live on every agent. Use format+checksum detection locally; transmit **type + confidence + count** only. Keyed EDM for high-entropy composite records, key server-side only. | firm |
| **D11** | **Embeddings and fuzzy fingerprints are content-equivalent.** Similarity-preserving hashes leak structure by construction; vec2text recovers ~92% of 32-token inputs exactly. Encrypted, access-controlled, **never distributed through the Hub**. Not a de-identification technique. | firm |
| **D12** | **Audit ledger = Postgres hash chain + forward integrity + external anchoring.** JetStream is a **bus only** — bounded retention is not a system of record. Key-evolving schemes so a post-compromise attacker cannot rewrite pre-compromise entries. Claim **"tamper-evident with forward-integrity between anchors"**; never "tamper-proof", which is unachievable in a single self-hosted trust domain. | firm |
| **D38** | **Completeness is provable only between witnessed anchors (T-019).** The hash chain and forward-secure signatures prove history was not ALTERED; they cannot prove it was not TRUNCATED — a root attacker can destroy the chain and rebuild a shorter, internally-consistent one. An `Anchor` is a witness-signed `(sequence, hash)` checkpoint of the ledger head; because the chain is linear, the head hash commits to the whole prefix, so no Merkle inclusion proof is needed. The `Witness` key MUST live in a trust domain the deployer does not control (a second host, WORM storage, a public transparency log) — an anchor the agent could forge proves nothing, since the agent is the party that might rewrite the log. `VerifyChain` matches each valid anchor to the chain, fails if a witnessed checkpoint is no longer satisfied (truncation of witnessed history is now DETECTED), and reports `AnchoredThrough` — completeness is ANCHORED only when a witness covers the last entry, else the prefix is proven and the tail stays UNVERIFIED. A forged anchor (wrong witness key) is ignored, never able to fail an honest chain. **Honest limits, stated where the interval is configured:** the undetectable-loss window equals the anchor interval (everything since the last anchor can still be truncated), and an anchor is only as trustworthy as the independence of its witness — Phase 1's witness is a separate keypair that proves the mechanism, not real third-party trust. | firm |
| **D20** | **Privacy-law features are Phase-1 architecture, not later additions:** enforced retention with automatic purge, purpose tagging at schema level, exclusion lists as a first-class policy primitive, pseudonymisation by default, audit trail of who *viewed* an investigation, four-eyes before HR-visible outcomes, a shipped DPIA template. GDPR Art. 35 makes a DPIA effectively mandatory; German works councils hold an **absolute veto** (BetrVG §87(1)(6)). | firm |
| **D47** | **View accountability is recorded by the write-capable control plane when it serves an investigation (T-013 seam, D20).** T-013 built `Ledger.RecordView` but the query CLI is a signer-less verifier that cannot append (D30); the write surface that serves fleet queries is the control plane (D41). `Server.View` serves an investigation's telemetry AND records the view — viewer, event, time — into `investigation_views` in one call, so obtaining the evidence and leaving a record are inseparable (the view is recorded FIRST: an attempted view is worth recording). An empty viewer is rejected — no unattributable view is silently kept — and callers prefix `unauthenticated:<os-user>` because there is still no authenticated OPERATOR identity (a sibling gap to agent identity T-017/D44; agent identity authenticates AGENTS, not the humans querying). `Views`/`ViewsBy` read it back — the "who looked at this" D20 requires. **Honest limits:** the view log is NOT the evidentiary ledger — no hash chain, no signatures, a compromised control plane could alter or omit it (same caveat as the aggregate, D41) — and the viewer is self-asserted (an OS identity), not verified. Loud-but-weak accountability beats an absent trail; overclaiming it as tamper-proof is forbidden. | firm |
| **D36** | **Retention purge tombstones rather than deletes (T-013), so GDPR erasure and the hash chain coexist.** Enforced retention (Art. 5/17) requires erasing expired personal data; the ledger is hash-chained, so deleting a row breaks every row after it. A purge instead erases the content columns (subject, decision payload, purpose) while KEEPING the skeleton — sequence, prev_hash, hash, sig, key_epoch. `VerifyChain` treats a tombstoned entry as an authenticated link: it still checks the prev-hash link and the signature over the stored hash, and ONLY waives the "hash matches current content" recompute, because the content is deliberately gone. `VerifyResult.Tombstoned` counts erased entries so an auditor sees erasure openly. Retention classes map to ages (short 30d / standard 365d / investigation HELD — a legal hold overrides routine purge; unspecified is treated as bounded, never an indefinite-retention hole). **Residual, stated:** a root attacker can tombstone+erase a LIVE entry to hide it, indistinguishable from a lawful purge without external anchoring — the SAME limit truncation already has (T-019), covered by the existing completeness caveat. **View accountability** (D20) is a `Ledger.RecordView` mechanism, built and tested; wiring it behind the query CLI is deferred to the write-capable control plane (T-023), because appending needs the signing key and the CLI is a pure verifier that must hold no signer (D30) — a discovery from implementation, not a dropped task. The four-eyes gate and employee-notice mechanism concern enforcement and are deferred to that phase; a DPIA template ships now (`docs/dpia-template.md`). | firm |
| **D23** | **UEBA supports both modes, gated.** Self-baseline computed locally, on by default. Peer-baseline is an **optional server-side Analytics module, off by default**, with its own consent/DPIA gate. Resolves the brief's silent contradiction between "Peer Analysis" and "the server coordinates, does not compute". Phase-1 cost is one field: a stable pseudonymous user ID. | firm |

## Runtime architecture

| # | Decision | Status |
|---|---|---|
| **D24** | **The endpoint pipeline is in-process and synchronous; NATS is the agent↔control-plane boundary only.** The brief names NATS as "the message bus" and says components communicate exclusively through events — read literally on the endpoint, that puts a broker inside the fanotify permission window, where [T-002](spike-t002-gc-pause.md) measured a 1-3µs typical / 532µs worst-case budget with a real process blocked in `TASK_UNINTERRUPTIBLE`. A broker round trip does not fit, and local-first evaluation (D1) means it should not have to. Both mechanisms are "the event bus" in the brief's language; only the second is NATS. Enforced by CI: `internal/core` may not import a broker (`scripts/check-core-deps.sh`). | firm |
| **D41** | **The control-plane store is the fleet AGGREGATE view, distinct from and NOT carrying the agent ledger's evidentiary guarantees (T-023).** The server side was referenced everywhere (T-017, T-018, verification) and built nowhere. `internal/controlplane.Server` subscribes to the three telemetry subjects the transport publishes (`openshield.v1.events/classifications/decisions`) and persists what it receives to `fleet_telemetry`. It stores only boundary-safe telemetry BY CONSTRUCTION: the transport has no `LocalClassification` method (D10), so only the `ClassificationSummary` — type+confidence+count — can arrive, and there is no path by which content could. A malformed message is COUNTED (`DecodeFailures`), never a silent vanish and never a stalled subscription. **The aggregate is NOT the evidentiary ledger:** it has no hash chain and no forward-secure signatures, a compromised control plane could alter it, and the evidentiary record is the agent's local forward-secure ledger (D12/D30), externally anchored (T-019). `agent_id` is self-asserted until identity (T-017). It coordinates and observes — it does NOT distribute policy or control agents (D14, "the server coordinates, it does not control"); Phase-1 policy is a local file (D1). Proven end to end over an embedded NATS + real Postgres. | firm |
| **D44** | **Each agent has its own Ed25519 identity (never a shared secret), enrolled via a single-use short-TTL token, signing telemetry with a monotonic sequence the control plane verifies for gaps; identity is revocable per agent (T-017).** The control plane previously accepted a self-asserted agent_id (D41); review (A6) flagged a shared fleet secret as fleet-wide risk — one compromised agent equalling fleet compromise. Now: the agent generates its own keypair (private key never leaves the host); an admin issues a token stored only as a SHA-256 hash with an expiry; `Enroll` verifies it is present/unexpired/unused, records the identity, and BURNS the token in one transaction (single-use). A token is enrollment-only, not a signing key — a stolen token enrolls a bogus (revocable, visible) agent but cannot impersonate an existing one. Telemetry is signed over a length-prefixed envelope with a monotonic sequence; `VerifySigned` rejects an unknown/revoked agent and a bad signature, ACCEPTS-and-records a sequence GAP (suppression between agent and control plane — the audit log's evidentiary reasoning: a trail that cannot reveal suppression is not evidentiary), and REJECTS a replay/reorder (seq ≤ last). `Revoke` is per-agent, so containing one endpoint does not touch the fleet. **Honest limits (D16):** root on the host reads the private key and can sign anything the agent could — the guarantee is attributable-and-revocable, not unforgeable; a gap is a signal (lossy network vs suppression), not an accusation; the key is in-memory (disk persistence for restart is deployment); and mTLS (authenticating the CONNECTION) is a complementary transport-layer concern, not this (which is the evidentiary application layer). | firm |
| **D25** | **Per-stage deadlines are owned by the dispatcher, not the stage.** A stage that sets its own deadline can set it to infinity, and an unbounded stage is the mechanism by which the responder hangs a machine. **Honest limit:** Go cannot preempt an uncooperative function, so the deadline bounds the dispatcher's *wait* and a pathological stage's goroutine may outlive it. That is why T-011's fail-open watchdog — which answers the kernel regardless of what the pipeline is doing — is an independent mechanism rather than a consequence of this one. | firm |

| **D26** | **The zero-core-change claim is narrowed, and the CI fitness test is necessary but not sufficient.** Capabilities of the same *shape* as existing ones need no core change; a new *shape* of data flow needs a small one. Demonstrated by [T-004](design-t004-peer-ueba.md), which also produced a worked example of the fitness test being gameable: letting Policy query the analytics store directly passes CI with zero core diffs while destroying the stage isolation the test exists to protect. Green CI is not evidence the architecture held. **BUILT (T-014):** `internal/fitness` adds a connector+stage+enforcer from outside core and flows an event to a Decision (proof a capability needs no core edit); `scripts/check-capability-boundary.sh` fails the build if core imports any capability package; and `TestFitnessTestKnowsItsLimits` guards this very caveat against silent deletion. Still necessary-not-sufficient. | firm |
| **D27** | **`Decision` carries a `context_version`.** Added before any consumer existed, because replay cannot reproduce a Decision without knowing which enrichment context applied, and retrofitting a field into a hash-chained ledger (T-009) means a migration and a break in chain continuity. Empty in Phase 1. | firm |

| **D28** | **Context is a resolved value on `State`, not an accessor, and a closed typed set, not a map.** An accessor would let Policy perform an unbounded lookup inside the permission window D24 protects; an open bag would be a surface through which a compromised control plane could influence decisions by inventing keys — D14's threat by another door. Absent enrichment MUST fail explicitly, never default: a defaulted risk score reads as "safe" and turns an analytics outage into a silent fail-open. Design: [`design-t030-context.md`](design-t030-context.md). Seam implemented, subsystem deliberately not. | firm |

## Security of the platform itself

| # | Decision | Status |
|---|---|---|
| **D13** | **Privilege-split agent; untrusted parsing in an unprivileged sandboxed worker.** Root + `CAP_SYS_ADMIN` + parsing attacker-controlled PDFs is the textbook way to make an org *less* secure — cf. ClamAV **CVE-2025-20260**, a PDF-parser heap overflow to RCE. The privileged process never decodes attacker-controlled bytes. Decompression-bomb limits before any parser runs. | firm |
| **D45** | **The privilege split is enforced by the systemd units, not left to the deployer (T-027).** A boundary that depends on a deployer remembering to drop capabilities erodes to everything-as-root. `deploy/systemd/openshield-agent.service` runs root but bounds capabilities to **only `CAP_SYS_ADMIN`** (fanotify); `openshield-worker.service` runs as a dedicated non-login user with an **empty capability set** plus the seccomp (T-012) and cgroup hardening drop-in (D35); `openshield-server.service` is an ordinary unprivileged service. `deploy/install.sh` is idempotent — binaries, the `openshield`/`openshield-worker` system users, units and drop-ins, `daemon-reload`, enable — refuses to run without root, and does NOT auto-start the agent (fanotify on an unconfigured host is the operator's call). **Upgrade is watchdog-safe (D18):** rebuild → install.sh → `systemctl restart openshield-agent`; a permission event during the restart window fails open (audited) rather than hanging a blocked process, which is why the watchdog was built even though Phase 1 does not enforce. Units are checked with `systemd-analyze verify` (directive-clean). **Not covered:** distro packages / signed artifacts (release engineering), TLS/secrets (deployment), non-systemd init (only Linux+systemd ships, D9); a full install-start cycle needs a systemd host + root and is a documented manual smoke test. | firm |
| **D35** | **The parser worker self-applies a seccomp network-deny filter before reading input (T-012).** Defense in depth on the half of the split that DOES parse attacker bytes (D13/D29): a parser RCE in a worker that cannot call `socket` cannot phone home, exfiltrate over the network, or open a reverse shell. Loaded with `NO_NEW_PRIVS` (no privilege needed) and TSYNC (covers every Go runtime thread; a thread-local filter is trivially bypassable), BEFORE the read loop (a filter applied after the first byte races a fast exploit). A **denylist** (network family + ptrace) in Phase 1, not a strict allowlist: an allowlist breaks whenever the Go runtime uses a new syscall, turning a runtime upgrade into a crash; the intended end state is an allowlist, and this delivers the property that matters now. Off Linux (dev only, D9) `Apply` returns `ErrUnsupported` and the worker warns LOUDLY — an absent sandbox is never a silent pass. Paired with a decompression-bomb guard (ratio/absolute-size/depth, rejected before the bytes reach a parser) and supervisor-enforced cgroup limits (`MemoryMax`, `MemorySwapMax=0`, `CPUQuota`, `TasksMax` in a systemd drop-in — a process cannot self-limit its cgroup without privilege, so that is configuration, not Go theatre). Verified: `socket()` returns EPERM after `Apply()`, including from another thread. | firm |
| **D15** | **Hub uses ed25519 per-author signing**, keys pinned at install, metadata cached with expiry and **fail-closed when stale**. Content-hash-against-index (CrowdSec's model) answers "did I download what the index said", not "who may publish" — and packs execute at high privilege on every endpoint, making this a fleet-wide root-equivalent supply chain. **Offline revocation is fundamentally unsolved**: the propagation window equals the sync interval. Document it; do not imply otherwise. | firm (design deferred) |

| **D29** | **The privilege split is two BINARIES, not one binary with a flag.** A single binary carries the parsers in its dependency graph regardless of which code path runs, which makes the import check — the only mechanism keeping this boundary real — prove nothing. Also stronger than D13 as written: the privileged process never *holds* attacker-controlled content, not merely never parses it. The worker opens files with its own unprivileged credentials and returns detector types, confidences and counts; `LocalClassification`, which holds matched text, never leaves the worker. Enforced by `scripts/check-agent-deps.sh`. | firm |
| **D48** | **The endpoint runs as THREE components — the privileged fanotify agent, the unprivileged network-capable ENGINE, and the sandboxed parser worker — and the three-process shape is forced by the constraints, not chosen (walking skeleton).** Assembling the pipeline surfaced that policy+audit can live in neither the privileged agent nor the worker: OPA uses `encoding/json`, which `check-agent-deps` BANS from the privileged agent (D29), and the audit ledger needs Postgres (network), which the worker's seccomp filter DENIES (D35). So `internal/engine` (unprivileged, network-capable) holds OPA and pgx and runs classify-via-worker → policy → decide → audit with the structured logger (D43 — closing the third deferred seam); the privileged agent holds neither OPA nor a parser (D29 stays enforced); the worker holds parsers but no network (D35 stays enforced). The classify stage bridges the worker's detector hits to `State.Classification` carrying type + confidence + count with EMPTY matched text — no content crosses the IPC (D29) — and a worker error is an auditable failure, never a silent clean result (D17). Proven end to end: a file with a seeded CPF flows through the REAL worker binary → policy → decision → REAL Postgres ledger, landing a verifiable entry. `cmd/openshield-engine` runs it (reloading its signer for write-resume, D46). The fanotify front-end forwarding events to the engine is a thin, privilege-gated step covered by the watchdog (T-011) and spike (T-005); observe-only (D1), it records, it does not enforce. | firm |

| **D30** | **Ledger forward integrity is an evolving Ed25519 KEYPAIR, not a symmetric ratchet — and neither the hash chain nor key evolution suffices alone.** The chain makes an edit detectable but only forces an attacker to rewrite the *tail*; key evolution decides where that rewritable tail *begins*. Together, and only together, they bound tampering to the moment of compromise. The scheme is asymmetric because a symmetric ratchet fails a requirement that is structural, not cryptographic: **verification needs the seed, and the seed forges.** The only party able to verify the log would be the only party able to fake it — which centralises fleet-wide forgery in the control plane, stops an endpoint verifying its own log, and collapses entirely on a single-node air-gapped deployment where no second trust domain exists. Each epoch generates a fresh keypair, publishes its public half signed by its predecessor, and destroys the predecessor's private key; verification walks the public chain from an out-of-band anchor and takes **no secret at all**. **This replaced a shipped implementation that provided no forward integrity whatsoever** — it retained a master seed for the process lifetime and derived every key from it, and the test that "proved" forward integrity passed only because it handed the attacker a derived key rather than the seed the attacker would actually have. Recorded here because the failure was in the threat model, not the code. **Honest limits:** key destruction in Go is best-effort (the GC may retain copies); everything inside the current epoch is forgeable by whoever takes the host; and internal consistency is not completeness — between anchors (T-019) a root attacker can destroy the chain and rebuild a shorter one that verifies perfectly, which is why `Verify` returns a structured result reporting completeness as UNVERIFIED rather than a bare boolean. | firm |
| **D31** | **The agent needs a reachable database to record anything, and there is no offline durability yet.** `Append` fails when Postgres is unreachable, the dispatcher surfaces that error to the caller as `ErrNotRecorded`, and nothing buffers, retries or degrades to a log line. That is deliberate: an entry silently dropped is indistinguishable from an event that never happened, which is the exact failure the ledger exists to prevent, so the gap is made loud rather than papered over. **The operational consequence, stated plainly: a laptop off the network produces no audit trail for that period.** For a product whose honest claim is *detection, not prevention*, this is currently its largest hole — offline store-and-forward is **T-024** and is unresolved until then. Deployments before T-024 lands should treat continuous database reachability as a requirement, not a convenience. | firm |
| **D40** | **The agent store-and-forwards telemetry through a bounded, durable, FIFO disk queue (T-024); overflow drops OLDEST and is a high-severity audit event.** "Offline-capable" is a core principle (D1) and nothing implemented it — a laptop off the network lost every Event it produced. `internal/transport/queue.QueueingTransport` wraps any `core.Transport` (callers unchanged; the seam was shaped for this, D24): reachable + empty queue → publish directly; unreachable OR queue non-empty → persist to disk and return success (the payload is durably held). Each payload is one atomically-written (temp-then-rename) file named by a monotonic sequence, so a crash mid-write cannot corrupt the spool and restart resumes exactly. `Flush` drains strict FIFO and STOPS at the first `ErrUnreachable`, keeping the undelivered tail — an out-of-order or truncated audit trail is a broken one. **The honest guarantee is "no SILENT loss", not "no loss":** the queue is bounded (an unbounded spool is a disk-exhaustion DoS), and on overflow it drops the OLDEST payload and fires a callback the agent wires to a high-severity audit entry (D17 — a silent drop is indistinguishable from nothing happening). Drop-oldest is deliberate: an overflowed evidence queue has already lost its guarantee, and the freshest activity is most likely to matter to an active investigation. **Limits stated:** survives a crash and reboot, not `rm -rf` or a dead disk; strict FIFO, no priority; payloads not encrypted at rest; this is the telemetry path, NOT the audit ledger's own offline durability (still [D31]'s open gap). | firm |
| **D32** | **The public-key chain is part of the ledger, not the signer's memory.** Forward integrity (D30) is only verifiable if the epoch public keys are available to a verifier; until T-010 they lived solely in the in-process `Signer`, so a second process could not verify and a restart with a fresh signer orphaned every prior entry — D30's "verification takes public material only" was true of the algorithm and false of the system. The chain is now persisted in `key_epochs` (public material only; no private column, asserted by test), each epoch written in the same transaction as the first entry that uses it, with a foreign key from `audit_entries.key_epoch` so an entry can never reference an unstored epoch. `Verify` loads the chain from the database and takes a caller-supplied **expected anchor**: given one it fails if the stored chain does not start there; given none it checks internal consistency only and reports completeness UNVERIFIED, naming the absent anchor — the difference between trusting the database to describe itself and not. Surfaced, like the D30 bugs, only when the CLI became the first honest cross-process consumer. **Honest limits:** writing cannot resume with a different signer (that would fork the chain under a new anchor; refused at `Open`), because surviving key material across a restart is T-017; and an anchor exported from a host that could later be compromised only attests to history if it was captured while the host was trusted (T-019 is the real fix). | firm |
| **D46** | **The ledger signer's CURRENT state is exportable/reloadable so an agent resumes its own chain across a restart (T-009 seam).** The public-key chain was persisted (D32) but the current private key lived only in memory, so a restart brought a fresh signer with a new anchor — `postgres.Open` refused it (`ErrCannotResumeWriting`) or a fork resulted. `Signer.Export()`/`core.LoadSigner` serialise the CURRENT epoch's private key, the epoch counter and the public chain; a reloaded signer has the identical anchor, so resume proceeds and appends continue ONE verifiable chain. Only the current epoch's private key is persisted — never a destroyed one — so forward security is intact: the current epoch is already the compromise window, and earlier keys are absent. The blob is integrity-checked (SHA-256 prefix — accidental corruption fails to load, not a MAC) and its private key is validated against the chain's current epoch (a signer never signs under a key the chain omits). `SaveSignerFile` writes it atomically at **mode 0600**; at-rest protection is filesystem permissions + the agent user, defeated by host root exactly as reading process memory is (D16) — encryption at rest (TPM/operator secret) is a noted follow-up that raises the bar against offline disk theft only. The agent's identity key (T-017/D44) is a distinct sibling gap, persisted the same way separately. | firm |
| **—** | **The `openshieldctl` CLI states the trust it does not have.** It is a read surface that holds no signer and can forge nothing (the D30 asymmetry). Until T-017 it authenticates no operator and records no viewer, so it deliberately writes **no** "who viewed this investigation" entry — an unattributable view record would misrepresent the D20 accountability trail as present. It runs for anyone who can reach the database, and it cannot prove completeness without an external anchor. All three limits are printed on the surface, not buried in docs. | firm |
